# Webhook

The webhook adapter exposes a generic JSON HTTP interface for internal systems.

```ts
import { webhook } from "@hunvreus/heypi";

webhook({
	secret: process.env.HEYPI_WEBHOOK_SECRET!,
	port: 3000,
	host: "127.0.0.1",
	path: "/webhook",
	replyHosts: ["internal.example.com"],
});
```

Webhook uses Node's HTTP server. It is not a Cloudflare Workers or Fetch adapter.

By default, webhook binds to `127.0.0.1`. Set `host: "0.0.0.0"` only when an external proxy, gateway, or firewall is meant to expose it.

Webhook is inbound-only. It does not implement adapter `send()`, so scheduled jobs cannot target webhook adapters.

## Auth

Requests must include either:

```text
authorization: Bearer <secret>
```

or:

```text
x-heypi-secret: <secret>
```

Use a long random secret. Treat it like an API token.

For production, put the webhook behind a reverse proxy or gateway with rate limiting. heypi also has a local in-process `maxInFlight` cap, but it is not a distributed provider-wide limiter.

## Limits

Defaults:

- `maxBodyBytes: 1_000_000`
- `maxInFlight: 32`
- `host: "127.0.0.1"`

Webhook payloads are for short JSON messages, not file uploads. Use provider attachments or a custom attachment store for files.

## Start A Thread

Omit `threadId` to create a new server-side thread:

```bash
curl -X POST http://localhost:3000/webhook/messages \
  -H "authorization: Bearer $HEYPI_WEBHOOK_SECRET" \
  -H "content-type: application/json" \
  -d '{"user":"alice@example.com","text":"Start incident review"}'
```

Response:

```json
{
	"ok": true,
	"threadId": "whth_...",
	"runId": "...",
	"status": "running"
}
```

`threadId` is opaque and generated by heypi. Store it on the caller side and use it for follow-ups.

## Follow Up

```bash
curl -X POST http://localhost:3000/webhook/threads/<threadId>/messages \
  -H "authorization: Bearer $HEYPI_WEBHOOK_SECRET" \
  -H "content-type: application/json" \
  -d '{"user":"alice@example.com","text":"What changed?"}'
```

## Status

```bash
curl http://localhost:3000/webhook/threads/<threadId>/runs/<runId> \
  -H "authorization: Bearer $HEYPI_WEBHOOK_SECRET"
```

Run status is read from heypi's normal persisted thread, turn, and message records. A restarted process can still answer status requests as long as it uses the same store.

## Async, Callback, And Sync

Webhook is async-first. Message requests return `202` while the turn runs.

Pass `replyUrl` to receive the final run payload by callback:

```json
{
	"threadId": "whth_...",
	"user": "alice@example.com",
	"text": "Run the check",
	"replyUrl": "https://internal.example.com/heypi/callback"
}
```

Callback URLs must use an allowed host:

```ts
webhook({
  secret: process.env.HEYPI_WEBHOOK_SECRET!,
  port: 3000,
  replyHosts: ["internal.example.com"],
});
```

If `replyHosts` is empty or omitted, requests with `replyUrl` are rejected.

For short requests, pass `sync: true`. Sync requests wait only up to the configured timeout, then return the current run state.

```json
{
	"text": "status",
	"sync": true
}
```

## Approvals

Webhook uses the same text fallback commands as other adapters:

```text
approve <approval-id>
deny <approval-id>
status
status <call-id>
cancel <turn-id-or-trace>
```

When a turn is waiting for approval, the run response includes structured approval data:

```json
{
	"status": "pending_approval",
	"approval": {
		"id": "...",
		"callId": "...",
		"reason": "..."
	}
}
```

The caller owns any UI buttons. Send `approve <id>` or `deny <id>` back to the same `threadId`.
